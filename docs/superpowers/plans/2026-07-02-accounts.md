# Accounts (PRD 08) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add optional Google/Apple sign-in to InSave that reconciles a person's fragmented anonymous islands (Android, iPhone Safari, iPhone PWA) into one account library, without regressing the anonymous fast path.

**Architecture:** Better Auth mounted in the existing raw Cloudflare Worker at `/api/auth/*`, users/sessions in D1 alongside current tables. Signed-in requests derive their owner id from the verified session; anonymous requests keep using the unverified client `user_id` exactly as today. Login triggers a server-side, idempotent, non-destructive merge that re-points the device's anonymous rows onto the account id, resolving `(user_id, canonical_url)` collisions by field precedence.

**Tech Stack:** TypeScript, Cloudflare Worker (raw, no framework), D1, Vitest, Wrangler, Better Auth (D1 adapter), `@block65/webcrypto-web-push`.

**Design source:** `docs/superpowers/specs/2026-07-02-accounts-design.md`

## Global Constraints

- **Anonymous fast path MUST NOT regress** — capture, sync, cron reminders, collections, importance/deadline all keep working fully signed-out. Login changes ownership, not capability. (Spec §2.6, §5.)
- **Trust rule:** when a session is present, owner id = account id **from the session**; a body/query `user_id` is ignored (403 on mismatch). No session → owner id = client-supplied `user_id` (unverified, as today). (Spec §5.2.)
- **Merge MUST be idempotent and non-destructive** — no branch deletes a reel; the only deletes are redundant duplicates already coalesced into a survivor. (Spec §7.4.)
- **Server-owned reminder state is account-authoritative** on merge: `reminder_status, next_due_at, cycle_count, ignored_count, last_surfaced_at` are never overwritten by merge. (Spec §7.3.)
- **Providers:** Google + Apple only. No email/password, no magic-link. (Spec §3, §4.)
- **Dedupe:** `canonical_url` uniqueness is **user-scoped** `(user_id, canonical_url) WHERE canonical_url <> ''`. (Spec D3, §10.2.)
- **Session cookie:** HttpOnly, Secure, SameSite=Lax, long expiry (sticky login). (Spec §5.1.)
- **Migrations:** existing production anonymous rows MUST be preserved; the real user merges them by signing in on each island. (Spec §10.3.)

## Phasing overview

- **Phase 0 — Task Zero spike (HARD GATE).** Prove Better Auth + D1 + Google round-trip on the Worker. Decide Better Auth vs Auth.js. **Nothing in Phases 1–4 starts until this gate passes.** — *Detailed below.*
- **Phase 1 — Auth foundation.** Apple provider, session middleware, the trust rule on existing endpoints, sign-in/out UI, sticky sessions.
- **Phase 2 — Merge engine.** `/api/merge`, re-point + collision resolution, field precedence, idempotency, non-destructiveness.
- **Phase 3 — iOS onboarding + push re-association.** `ios.html` two-context truth; push subs follow the merge.
- **Phase 4 — Account deletion + release hardening.** Delete-account, production OAuth config, on-device session-durability verification.

> **Post-gate expansion:** Phases 1–4 are captured as a task-level roadmap at the end of this document. They are intentionally **not** written as exact-code TDD yet: the precise Better Auth API surface (session object shape, handler signatures, cookie plumbing) is unknown until the Phase 0 spike locks the stack. Immediately after the gate passes, expand the roadmap for the chosen stack into exact-code TDD tasks, following this same task structure.

## Post-gate roadmap (Phases 1–4)

> Expand each task below into exact-code TDD steps **after** the Phase 0 gate, using the observed session/user shape and real auth route paths from Task 0.6. Structure each task as: Files / Interfaces / failing test → verify fail → minimal impl → verify pass → commit. Follow existing repo test patterns (Vitest + `fake-indexeddb`).

### Phase 1 — Auth foundation

- **Task 1.1 — Owner-id resolver (the trust rule).** Create a single `resolveOwner(request, env, body)` helper: if a valid session cookie is present, return `{ ownerId: session.user.id, authed: true }`; else return `{ ownerId: body/query user_id, authed: false }`. When authed and a body `user_id` is present and differs, signal a 403. Unit-test both branches + the mismatch. Files: new `worker/owner.ts` (or into `worker/auth.ts`), test alongside. This is the release-critical piece (spec §5.2) — test it hardest.
- **Task 1.2 — Apply the resolver to existing endpoints.** Thread `resolveOwner` into `handleSync`, `handlePull`, `handleCollections`, `handleSubscribe`, `parseAction` in `worker/index.ts`, replacing direct trust of the body/query `user_id`. Regression-test that anonymous requests behave exactly as before, and that a signed-in request ignores a spoofed body id. Preserve the server-owned column exclusion in `UPSERT_SQL` (`worker/index.ts:74-85`).
- **Task 1.3 — Add the Apple provider.** Extend `createAuth` with the Apple social provider; handle Apple's one-time name/email-on-first-auth by persisting it on first sign-in. Confirm the Apple SIWA obligation (spec §13). Manual round-trip like Task 0.4 for Apple.
- **Task 1.4 — Sign-in / sign-out UI + session state on the client.** Add sign-in (Google/Apple) and sign-out affordances; a small client `getSession()` wrapper; reflect signed-in state in the UI. No change to anonymous capture flows. Test session-state rendering.
- **Task 1.5 — Sticky session verification.** Confirm the durable cookie survives reloads and the app remaining closed; long expiry configured (spec §5.1).

### Phase 2 — Merge engine

- **Task 2.1 — User-scoped dedupe index migration.** Replace global `idx_canonical_url` (`schema.sql:64-66`) with unique `(user_id, canonical_url) WHERE canonical_url <> ''`. Provide the forward migration (see Task 4.x migration convention) and update `handleSync` conflict handling (`worker/index.ts:178-189`) so an existing-row "accepted" response is per-user. Test: two users may now hold the same `canonical_url`; one user cannot duplicate it.
- **Task 2.2 — `POST /api/merge` endpoint (authed).** Derive `account_id` from the session (never body); take the body's anon `user_id` as the absorb source. Reject if unauthenticated. Test auth gating.
- **Task 2.3 — Re-point (no-collision path).** For `pending_capture`/`collections`/`push_subscriptions` anon rows whose `(account_id, canonical_url)` does not already exist, `UPDATE ... SET user_id = account_id`. `user_settings`: adopt anon row only if the account has none. Test each table.
- **Task 2.4 — Collision resolution (field precedence).** For `pending_capture` rows colliding on `(account_id, canonical_url)`: coalesce device-owned content into the surviving account row (additive; `tagged_at` breaks genuine conflicts; `deadline_at` keeps whichever set), **never** touch the five server-owned reminder-state columns, then delete the redundant anon row. Test: account reminder state preserved; anon-only tags/collection/deadline adopted; conflicting tags resolve by newer `tagged_at`; no reel lost. **Validate the `tagged_at` tiebreak with a real merge test before locking (spec §13).**
- **Task 2.5 — Atomicity, idempotency, non-destructiveness.** Wrap a per-anon-id merge in a single D1 batch. Test: replay of the same merge is a no-op; interrupted/retried merge converges; no branch deletes a reel that wasn't a coalesced duplicate.
- **Task 2.6 — Client post-merge swap.** After a successful merge, the client swaps its IndexedDB `user_id` (`src/db.ts:44-51`) to the account id and re-pulls (`src/reminder-pull.ts`). Update the call-sites listed in spec §14. Test the swap + re-pull reconcile.
- **Task 2.7 — Cron sees re-pointed ids.** Confirm `worker/cron.ts:14-20` grouping now finds items under the account id and that orphaned anon ids generate no cycles. Test cron grouping post-merge.

### Phase 3 — iOS onboarding + push re-association

- **Task 3.1 — `ios.html` two-context truth.** Add the "paste = no account" vs "Shortcut = sign in on both app and Safari" copy as a real, understandable design element (spec §8, PRD §7.5). Drive via `src/ios-onboarding.ts`.
- **Task 3.2 — Push subscription follows the merge.** Verify `push_subscriptions` re-points during merge (Task 2.3) and that `ensureSubscription()` re-registers under the account id on next app open (`src/register-sw.ts:11`). Test: a subscription follows the merge; the dead anon id receives no sends (`worker/notify.ts`).

### Phase 4 — Account deletion + release hardening

- **Task 4.1 — Migration convention.** Introduce a repeatable, ordered remote-migration path (numbered migration files applied via `wrangler d1 execute --remote`, or an equivalent documented sequence) so the dedupe-index change and Better Auth tables reach prod without clobbering the real islands (spec §10.3). This underpins Task 2.1's remote application.
- **Task 4.2 — `DELETE` account.** Remove Better Auth records **and** the account's owned rows in `pending_capture`/`collections`/`user_settings`/`push_subscriptions WHERE user_id = account_id` (spec §10.4). Test: deletion removes exactly the account's data, leaves other owners untouched. Data export deferred (non-goal).
- **Task 4.3 — Production OAuth config.** Add production redirect URIs + consent config for Google & Apple; production `AUTH_BASE_URL`; secrets via `wrangler secret`.
- **Task 4.4 — On-device session durability.** Verify iOS PWA session persistence on-device (the flaky bit, spec §10, §13); adjust cookie/expiry settings if needed.
